# 需求挖掘项目需求文档

## 1. 项目目标

每天自动采集 10-20 条具备个人产品开发潜力的需求线索，并按课程方法进行轻量分析，最终沉淀成可检索、可复盘、可继续深挖的机会库。

本项目不是新闻摘要器，也不是泛泛的趋势日报。它的目标是发现“能做产品”的线索，尤其是：

- 新词刚出现、搜索需求开始上升、供给还没跟上的新词站机会
- Hugging Face Trending 上出现的新模型、新能力、新项目
- 用户公开抱怨暴露出的真实痛点
- 市场里已存在但供给太慢、太贵、太复杂、太烂的供需失衡
- 可以在 48 小时内做出 MVP 承接的 Alpha 窗口

## 2. 方法论拆解

### 2.1 幸福的方式：收集抱怨

核心原则：不要问用户想要什么，要收集用户哪里不爽。

优先采集这些信号：

- “太难用”“有没有替代品”“求平替”“避雷”“吐槽”
- 竞品评论区的一星/差评
- Reddit、X、Product Hunt、GitHub Issues、插件商店里的重复抱怨
- 工具链断点：导入、导出、迁移、部署、支付、协作、中文/英文支持

筛选问题：

- 这个抱怨是否反复出现？
- 抱怨者是一类人，还是单个偶发个体？
- 现有产品为什么没有解决好？
- 是否能形成一个小工具、模板、工作流、插件或服务？

### 2.2 功利的方式：找供需失衡

核心原则：需求已经存在，供给还不够好。

供需失衡类型：

- 搜索词有需求，但 SERP 供给弱
- 新产品/新政策/新平台变化导致用户工作流被打断
- 新技术很强，但普通用户不会用
- 大厂产品太重、太贵、太慢，小团队可以做 20% 聚焦版本
- Fiverr / Upwork 热门人工服务可以被产品化

筛选问题：

- 为什么有人需要它？
- 这些人是谁？
- 他们现在怎么解决，为什么不爽？
- 我能把哪个环节做得更快、更便宜、更简单？
- 这个需求是一次性，还是可以持续滚大？

### 2.3 Alpha 意识：找窗口期

核心原则：不要问“值不值得做”，要问“这里有没有 Alpha”。

Alpha 判断维度：

- 信息差：是否比中文二手信息更早？
- 行动速度：是否能在 48 小时内做出 MVP？
- 判断力：三个月后是否还有人用？需求是否真实？供给是否不足？

线索如果只“有趣”但无法行动，不进入 Top 10。

### 2.4 新词站：承接高意图搜索

核心原则：新词不是关键词，它首先是一个临时入口。

优先采集：

- 新模型、新产品、新项目、新 API、新开源库
- 搜索需求正在上升但 SERP 结果还弱的新词
- 带有 `generator`、`tool`、`template`、`maker`、`online`、`free`、`alternative`、`tutorial`、`API` 等后缀的任务词
- 官方供给贵、难、慢、不方便的词

新词站判断问题：

- 这个词是否刚出现，热度是否上升？
- 搜这个词的人是想“用”，还是只想“看”？
- 官方供给是否贵、难、慢、不方便？
- 是否能在 48 小时内做出可承接 MVP？
- 背后是否有订阅、credits、转售、服务层、工具层空间？
- 是否有商标/版权风险？

## 3. 每日采集配比

默认 10-20 条线索建议配比：

- 约 40%：新词站机会
- 约 30%：Hugging Face Trending / GitHub 新技术机会
- 约 30%：新闻窗口、AI 产品发布、抱怨 / 差评 / 社群痛点、服务产品化 / 搜索词机会

数量规则：

- 每天至少 10 条
- 高质量线索充足时最多扩展到 20 条
- 不为了凑满 20 条放入低质量内容
- 如果某天 Hugging Face 或新词机会特别强，可以动态提高这两类占比

## 4. 采集来源

第一优先级：

- Hugging Face Trending
- GitHub Trending
- Product Hunt
- Hacker News
- Google Trends
- AI 产品新闻源

第二优先级：

- Reddit：`r/LocalLLaMA`、`r/SideProject`、`r/SaaS`、`r/ArtificialInteligence`、`r/MachineLearning`
- X / Twitter 公开讨论
- Fiverr / Upwork 热门服务
- 插件商店、SaaS 评论区、GitHub Issues

中文源作为补充：

- 小红书、知乎、微博、即刻、公众号、行业社群公开内容

## 5. 评分模型

每条线索使用 5 个维度评分，每项 1-5 分：

- 需求强度：用户是否明确要解决问题
- 供给缺口：现有供给是否明显不足
- Alpha 时效：是否处于早期窗口
- MVP 可行性：48 小时内能否做出第一版
- 变现潜力：是否能订阅、credits、一次性收费、服务化或导流

总分解释：

- 21-25：高优先级，值得单独深挖
- 16-20：进入机会池，持续观察
- 11-15：记录但暂不行动
- 10 以下：除非有特殊原因，否则丢弃

## 6. 每日报告格式

每日 10-20 条，每条线索控制在 3 分钟内看完，包含：

- 线索标题
- 来源和链接
- 类型：新词站 / HF Trending / 新闻窗口 / 抱怨 / 搜索词 / 服务产品化
- 一句话需求
- 用户是谁
- 供需失衡
- 可做 MVP
- 变现方式
- 风险
- 评分
- 是否建议深挖

## 7. GitHub 推送策略

每日生成：

- `daily/YYYY-MM-DD.md`
- 追加或更新 `data/opportunities.csv`

每日成果需要推送到 GitHub，便于后续收集和复盘需求。

提交范围：

- 允许提交：`daily/`、`data/`
- 可选提交：`README.md`、`docs/`、`templates/`
- 禁止提交：04、05、06、07 等课程原文 Markdown

提交信息格式：

```text
daily: add opportunities YYYY-MM-DD
```

执行策略：

```bash
git add daily data
git commit -m "daily: add opportunities YYYY-MM-DD"
git push
```

注意：不要使用 `git add .`，避免把课程原文推送到 GitHub。

如果当前仓库尚未配置 GitHub remote，则每日任务仍应生成本地日报，并在日报末尾记录推送未完成的原因。

## 8. DeepSeek 接入计划

第一版不使用模型，先以规则采集和人工可读报告为主。

第二版可接 DeepSeek 做：

- 摘要
- 去重
- 评分解释
- 新词变体扩展
- MVP 方案生成

安全规则：

- API key 不写入代码
- 使用环境变量 `DEEPSEEK_API_KEY`
- `.env` 必须加入 `.gitignore`
