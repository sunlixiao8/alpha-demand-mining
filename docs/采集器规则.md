# 每日采集器规则

## 总原则

每天采集 10-20 条，但必须足够像“产品机会”，不是普通新闻。

日报不是“链接堆砌”，而是商业决策简报。每条线索都要回答：

- 结论：今天要不要深挖？
- 证据：我凭什么认为它有机会？
- 为什么现在：窗口期在哪里？
- 用户是谁：谁会真的痛？
- 真实需求：用户要完成什么动作？
- 供需失衡：现有方案哪里慢、贵、难、差？
- 切入角度：个人/小团队从哪里切？
- MVP：48 小时内做什么验证？
- 分发路径：第一批用户从哪里来？
- 变现方式：怎么收钱？
- 风险：为什么可能不成立？
- 下一步验证：明天具体做什么？
- 置信度：证据够不够？

采集时优先问：

1. 这里有没有新词？
2. 这里有没有高意图搜索？
3. 这里有没有 Hugging Face / GitHub 新能力？
4. 用户是否正在抱怨？
5. 供给是不是太慢、太贵、太复杂、太难用？
6. 48 小时内能不能做 MVP？

## 类型优先级

### P0：新词站机会

强信号：

- 新模型、新能力、新产品刚发布
- 词本身开始被反复提及
- 用户在搜“哪里用”“free”“generator”“tutorial”“alternative”
- 官方入口不方便、地区受限、价格高、部署麻烦

必须标记：

- 是否适合 EMD
- 是否有商标风险
- 是否适合做工具页、教程页、替代入口、API 代理、资源导航

### P0：Hugging Face Trending

强信号：

- 新模型能力明显改变某个任务成本
- Demo 很强但普通用户不易用
- 开源项目缺少产品化包装
- 评论区、社区、模型卡里能看到具体使用场景

必须转译成：

- “谁会用”
- “用来完成什么任务”
- “现有方案哪里麻烦”
- “能不能做一个更简单的 Web 工具”

### P1：新闻窗口

强信号：

- 平台政策、模型发布、产品限制、地区限制、价格变化
- 用户工作流被打断
- 两三天内可以做迁移、转换、教程、生成、代理、聚合工具

### P1：抱怨线索

强信号：

- 同一抱怨出现 3 次以上
- 用户主动求替代品
- 用户已经为旧方案付费但仍不满意

### P2：服务产品化

强信号：

- Fiverr / Upwork 服务订单多
- 交付物标准化
- 60%-80% 流程可自动化

## 排除规则

直接排除：

- 抓取失败或链接不可访问
- 没有描述、没有上下文、无法判断用户任务
- 只有热闹，没有明确动作
- 纯融资、纯大厂战略、纯观点争论
- 用户只想“了解一下”，没有工具/服务意图
- 明显高法律风险且无法转向正常产品
- 需要大型团队长期研发，48 小时完全无法验证

谨慎标记：

- 商标词 EMD
- 医疗、金融投资、成人、灰产
- 依赖不可控封号/绕限制路径的产品

## 自检规则

生成日报前必须自检：

- 是否每条都出现了不同的“切入角度”和“MVP”
- 是否保留了原始信号，方便人工复核
- 是否有明确分发路径，而不是只说“做个工具”
- 是否能支持商业决策：深挖、观察、丢弃
- 如果前 5 条读起来像同一模板，说明报告不合格，需要重写

自动化自检由 `scripts/audit_report.py` 执行。它会检查：

- 线索数量是否在 10-20 条
- 每条是否包含完整商业决策字段
- 是否存在坏链接
- 是否存在模板化短语大面积重复
- 是否有明确的深挖/观察/丢弃建议

## 每日排序规则

先按总分排序，再人工保留多样性：

- 每天至少 10 条
- 高质量线索充足时最多 20 条
- 新词站机会约 40%
- Hugging Face / GitHub 技术机会约 30%
- 新闻窗口、抱怨、服务产品化、搜索词机会约 30%

如果没有足够高质量线索，宁可停在 10 条左右，也不要硬凑垃圾机会。
